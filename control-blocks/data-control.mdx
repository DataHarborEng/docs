# Data Control

**Privacy controls, not privacy code.**

Field-level privacy controls that protect sensitive data without requiring engineering effort. Configure once, apply everywhere.

<Info>
  Controls are an ordered pipeline that can include both privacy filters and [data transforms](/control-blocks/data-transform). You don't need transforms ã you can use filter controls on their own. This page covers privacy filter controls.
</Info>

<Tip>
  For a complete Virtual API Configuration, start with `version: "0.3"` and define controls under `objects.<objectName>.controls`. Snippets below that start with `controls:` are fragments meant to live inside an object definition.
</Tip>

## Object Definitions

Data controls are defined per object type in your Virtual API Configuration. The `objects` key maps resource names to their control configurations.

Given an API with these endpoints:
- `/properties` ã list of properties
- `/properties/{propertyId}/inspections` ã inspections for a property
- `/properties/{propertyId}/inspections/{inspectionId}/issues` ã issues for an inspection

This configuration defines controls for each object type:

```yaml
version: "0.3"
objects:
  properties:    # matches /properties, /properties/{propertyId}
    controls:
      - type: redact
        fields: [ownerFirstName, ownerLastName]
  inspections:   # matches .../inspections, .../inspections/{inspectionId}
    controls:
      - type: redact
        fields: [inspectorFirstName]
  issues:        # matches .../issues, .../issues/{issueId}
    controls:
      - type: redact
        fields: [title]
```

Each control has two required properties:
- `type` ã the operation to apply (`redact`, `tokenize`, `anonymize`, `mask`, `hash`, `allow`)
- `fields` ã one or more field names to apply it to

You can also add `required: true` when a missing field should fail the request instead of being skipped.

<Tip>
  You'll see `fields` written two ways in our examples ã both are valid YAML and produce identical results:
  ```yaml
  # Inline style
  fields: [first_name, last_name]

  # Multi-line style
  fields:
    - first_name
    - last_name
  ```
</Tip>

### The _default Object

`_default` is an optional catch-all for objects not explicitly defined.

```yaml
version: "0.3"
objects:
  properties:
    controls:
      - type: redact
        fields: [ownerName]
  _default:
    controls:
      - type: redact
        fields: [createdBy, modifiedBy]
```

**Precedence rules:**

| Scenario | Behavior |
|----------|----------|
| Request matches undefined object | `_default` controls apply |
| Request matches named object | Only the named object's controls apply. `_default` is not merged. |

## Control Types

### Redact

Remove sensitive data entirely. Strings become empty, numbers become zero.

```yaml
controls:
  - type: redact
    fields: [ssn, date_of_birth, drivers_license]
```

**Input:**
```json
{ "name": "Jane Doe", "ssn": "123-45-6789", "balance": 1000 }
```

**Output:**
```json
{ "name": "Jane Doe", "ssn": "", "balance": 1000 }
```

### Tokenize

Replace sensitive values with consistent, syntax-preserving tokens. The same input always produces the same token within the same Virtual API and matched object definition ã enabling analytics without exposing raw data.

```yaml
controls:
  - type: tokenize
    fields: [email, phone]
```

**Input:**
```json
{ "email": "jane@example.com", "phone": "555-1234" }
```

**Output:**
```json
{ "email": "tok_abc123def456", "phone": "tok_789xyz012" }
```

Tokenization is **scope-bound** ã the same input value always produces the same token within the same Virtual API and matched object definition. Different object definitions or different Virtual APIs can produce different tokens for the same input.

### Anonymize

Generate randomized replacements that preserve data structure. Each call produces a different value, making re-identification impossible.

```yaml
controls:
  - type: anonymize
    fields: [email, name]
```

**Input:**
```json
{ "email": "jane@example.com", "name": "Jane Doe" }
```

**Output (first call):**
```json
{ "email": "user_7f3a@anon.local", "name": "Alex Smith" }
```

**Output (second call):**
```json
{ "email": "user_9b2c@anon.local", "name": "Jordan Lee" }
```

### Mask

Partially redact values while preserving enough structure for context. Useful in support, debugging, and audit log scenarios where you need to understand the data shape without revealing full values.

```yaml
controls:
  - type: mask
    fields: [ssn, phone]
```

**Input:**
```json
{ "name": "John Smith", "ssn": "123-45-6789", "phone": 5551234567 }
```

**Output:**
```json
{ "name": "Jo** ***th", "ssn": "12*-**-**89", "phone": 67 }
```

Masking behavior varies by data type:

| Type | Rule | Example |
|------|------|---------|
| `string` | Keeps first & last N characters visible (N=2 for strings longer than 4 chars, N=1 otherwise). Strings of 2 or fewer characters are fully masked. Whitespace is preserved. | `"John Smith"` ã `"Jo** ***th"` |
| `double` | Zeros leading digits, keeps last 2 visible per segment | `123456.78` ã `000056.78` |
| `long` | Zeros leading digits, keeps last 2 visible | `5551234567` ã `0000000067` |
| `bool` | Always returns `false` | `true` ã `false` |

### Hash

Replace values with a one-way SHA-256 digest. Consumers can correlate and deduplicate records without seeing real values ã the same input within the same Virtual API and matched object definition always produces the same hash.

```yaml
controls:
  - type: hash
    fields: [email, customerId]
```

**Input:**
```json
{ "email": "jane@example.com", "customerId": 12345 }
```

**Output:**
```json
{ "email": "a3f2b8c1e9d7....", "customerId": 7293810456 }
```

Hash behavior varies by data type:

| Type | Output | Notes |
|------|--------|-------|
| `string` | 64-character hex digest | Full SHA-256 hex of `scope:input` |
| `double` | Deterministic double | Derived from hash bytes |
| `long` | Deterministic long | From first 8 hash bytes |
| `bool` | Deterministic boolean | Based on first hash byte |

<Info>
  Hashing is **scope-bound** ã different object definitions or different Virtual APIs produce different hashes for the same input value, preventing broad cross-context correlation.
</Info>

### Allow

Pass data through unchanged. Useful when you want to be explicit about which fields are permitted, even though fields not referenced by any control already pass through unchanged.

```yaml
controls:
  - type: allow
    fields: [publicName]
```

**Input:**
```json
{ "publicName": "Acme Corp" }
```

**Output:**
```json
{ "publicName": "Acme Corp" }
```

## Execution Rules

- Data Control uses dot notation in v0.3. Slash notation like `address/city` is rejected.
- Filter controls target scalar values only: strings, numbers, and booleans. If a path resolves to an object or array, the request fails instead of replacing the whole structure.
- If the matched payload is an array, the control list runs against each object element automatically. Non-object elements are skipped.
- Controls execute top-to-bottom. Later controls see values produced by earlier controls.
- Set `required: true` on a control when a missing field should fail instead of being skipped.

## Targeting Nested Fields

Use dot notation to target nested fields in v0.3:

```yaml
controls:
  - type: redact
    fields: [user.ssn, user.address.street, billing.card_number]
```

## Targeting Arrays

Apply controls to all items in an array:

```yaml
controls:
  - type: tokenize
    fields: ["[contacts].email", "[contacts].phone"]
```

If the matched payload itself is a root array, DataHarbor applies the same control list to each object element automatically.

## Combining Controls

Apply multiple control types within an object definition:

```yaml
objects:
  customers:
    controls:
      - type: redact
        fields: [ssn, date_of_birth]
      - type: tokenize
        fields: [email]
      - type: mask
        fields: [phone]
      - type: anonymize
        fields: [name]
      - type: hash
        fields: [customerId]
```

## When to Use Each

| Control Type | Use case |
|--------------|----------|
| **redact** | Data that should never leave your system (SSN, credentials) |
| **tokenize** | Identifiers needed for correlation/analytics without exposure |
| **anonymize** | Data for ML training, testing, or demos |
| **mask** | Support/debugging contexts where partial visibility is needed |
| **hash** | Correlation and deduplication without exposing original values |
| **allow** | Explicitly permit fields that should pass through unchanged |
