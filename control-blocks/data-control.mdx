# Data Control

**Privacy controls, not privacy code.**

Field-level privacy controls that protect sensitive data without requiring engineering effort. Configure once, apply everywhere.

<Info>
  Controls are an ordered pipeline that can include both privacy filters and [data transforms](/control-blocks/data-transform). You don't need transforms ÔÇö you can use filter controls on their own. This page covers privacy filter controls.
</Info>

## Object Definitions

Data controls are defined per object type in your Virtual API spec. The `objects` key maps resource names to their control configurations.

Given an API with these endpoints:
- `/properties` ÔÇö list of properties
- `/properties/{propertyId}/inspections` ÔÇö inspections for a property
- `/properties/{propertyId}/inspections/{inspectionId}/issues` ÔÇö issues for an inspection

The spec defines controls for each object type:

```yaml
objects:
  properties:    # matches /properties, /properties/{propertyId}
    controls:
      - type: redact
        fields: [ownerFirstName, ownerLastName]
  inspections:   # matches .../inspections, .../inspections/{inspectionId}
    controls:
      - type: redact
        fields: [inspectorFirstName]
  issues:        # matches .../issues, .../issues/{issueId}
    controls:
      - type: redact
        fields: [title]
```

Each control has two required properties:
- `type` ÔÇö the operation to apply (`redact`, `tokenize`, `anonymize`, `mask`, `hash`, `allow`)
- `fields` ÔÇö one or more field names to apply it to

<Tip>
  You'll see `fields` written two ways in our examples ÔÇö both are valid YAML and produce identical results:
  ```yaml
  # Inline style
  fields: [first_name, last_name]

  # Multi-line style
  fields:
    - first_name
    - last_name
  ```
</Tip>

### The _default Object

`_default` is an optional catch-all for objects not explicitly defined.

```yaml
objects:
  properties:
    controls:
      - type: redact
        fields: [ownerName]
  _default:
    controls:
      - type: redact
        fields: [createdBy, modifiedBy]
```

**Precedence rules:**

| Scenario | Behavior |
|----------|----------|
| Request matches undefined object | `_default` controls apply |
| Request matches named object | Controls merge. If the same field appears in both, the named object wins. Other `_default` controls still apply. |

## Control Types

### Redact

Remove sensitive data entirely. Strings become empty, numbers become zero.

```yaml
controls:
  - type: redact
    fields: [ssn, date_of_birth, drivers_license]
```

**Input:**
```json
{ "name": "Jane Doe", "ssn": "123-45-6789", "balance": 1000 }
```

**Output:**
```json
{ "name": "Jane Doe", "ssn": "", "balance": 1000 }
```

### Tokenize

Replace sensitive values with consistent, syntax-preserving tokens. Same input always produces the same token ÔÇö enabling analytics without exposing raw data.

```yaml
controls:
  - type: tokenize
    fields: [email, phone]
```

**Input:**
```json
{ "email": "jane@example.com", "phone": "555-1234" }
```

**Output:**
```json
{ "email": "tok_abc123def456", "phone": "tok_789xyz012" }
```

Tokenization is **deterministic per Virtual API** ÔÇö the same input value always produces the same token within that Virtual API. Different Virtual APIs produce different tokens for the same input.

### Anonymize

Generate randomized replacements that preserve data structure. Each call produces a different value, making re-identification impossible.

```yaml
controls:
  - type: anonymize
    fields: [email, name]
```

**Input:**
```json
{ "email": "jane@example.com", "name": "Jane Doe" }
```

**Output (first call):**
```json
{ "email": "user_7f3a@anon.local", "name": "Alex Smith" }
```

**Output (second call):**
```json
{ "email": "user_9b2c@anon.local", "name": "Jordan Lee" }
```

### Mask

Partially redact values while preserving enough structure for context. Useful in support, debugging, and audit log scenarios where you need to understand the data shape without revealing full values.

```yaml
controls:
  - type: mask
    fields: [ssn, phone]
```

**Input:**
```json
{ "name": "John Smith", "ssn": "123-45-6789", "phone": 5551234567 }
```

**Output:**
```json
{ "name": "Jo** ***th", "ssn": "12*-**-**89", "phone": 67 }
```

Masking behavior varies by data type:

| Type | Rule | Example |
|------|------|---------|
| `string` | Keeps first & last N characters visible (N=2 for strings longer than 4 chars, N=1 otherwise). Strings of 2 or fewer characters are fully masked. Whitespace is preserved. | `"John Smith"` Ôćĺ `"Jo** ***th"` |
| `double` | Zeros leading digits, keeps last 2 visible per segment | `123456.78` Ôćĺ `000056.78` |
| `long` | Zeros leading digits, keeps last 2 visible | `5551234567` Ôćĺ `0000000067` |
| `bool` | Always returns `false` | `true` Ôćĺ `false` |

### Hash

Replace values with a one-way SHA-256 digest. Consumers can correlate and deduplicate records across datasets without seeing real values ÔÇö the same input within the same Virtual API always produces the same hash.

```yaml
controls:
  - type: hash
    fields: [email, customerId]
```

**Input:**
```json
{ "email": "jane@example.com", "customerId": 12345 }
```

**Output:**
```json
{ "email": "a3f2b8c1e9d7....", "customerId": 7293810456 }
```

Hash behavior varies by data type:

| Type | Output | Notes |
|------|--------|-------|
| `string` | 64-character hex digest | Full SHA-256 hex of `scope:input` |
| `double` | Deterministic double | Derived from hash bytes |
| `long` | Deterministic long | From first 8 hash bytes |
| `bool` | Deterministic boolean | Based on first hash byte |

<Info>
  Hashing is **scope-bound** ÔÇö different Virtual APIs produce different hashes for the same input value, preventing cross-lease correlation.
</Info>

### Allow

Pass data through unchanged. Useful when you want to be explicit about which fields are permitted, or to override a `_default` control for specific objects.

```yaml
controls:
  - type: allow
    fields: [publicName]
```

**Input:**
```json
{ "publicName": "Acme Corp" }
```

**Output:**
```json
{ "publicName": "Acme Corp" }
```

## Targeting Nested Fields

Use dot notation to target nested fields:

```yaml
controls:
  - type: redact
    fields: [user.ssn, user.address.street, billing.card_number]
```

## Targeting Arrays

Apply controls to all items in an array:

```yaml
controls:
  - type: tokenize
    fields: ["[contacts].email", "[contacts].phone"]
```

## Combining Controls

Apply multiple control types within an object definition:

```yaml
objects:
  customers:
    controls:
      - type: redact
        fields: [ssn, date_of_birth]
      - type: tokenize
        fields: [email]
      - type: mask
        fields: [phone]
      - type: anonymize
        fields: [name]
      - type: hash
        fields: [customerId]
```

## When to Use Each

| Control Type | Use case |
|--------------|----------|
| **redact** | Data that should never leave your system (SSN, credentials) |
| **tokenize** | Identifiers needed for correlation/analytics without exposure |
| **anonymize** | Data for ML training, testing, or demos |
| **mask** | Support/debugging contexts where partial visibility is needed |
| **hash** | Correlation and deduplication without exposing original values |
| **allow** | Explicitly permit fields, or override `_default` controls |
